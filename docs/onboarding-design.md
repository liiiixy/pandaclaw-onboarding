# PandaClaw Onboarding — 方案设计文档

## 一、背景 & 动机

### 为什么需要 Onboarding？

PandaClaw 的定位是「AI 伙伴」而非「AI 工具」。工具型产品（ChatGPT、Cursor）用户打开就用，不需要建立关系；但伙伴型产品的核心体验是「关系感」——用户需要觉得这个 AI 认识我、懂我、有自己的个性。

Onboarding 是建立这层关系的第一个触点。它决定了：
- 用户是把 AI 当工具用，还是当伙伴处
- AI 有没有「人格」的第一印象
- 用户是否愿意持续回来

### 行业现状

大多数 AI 产品的 onboarding 是表单式的：选兴趣标签、填职业、勾选使用场景。这种方式高效但冰冷，适合工具不适合伙伴。

少数产品尝试对话式 onboarding（如 Character.ai、Replika），但往往存在两个问题：
1. **漫无目的**：聊了很久但没有收集到有用信息，也没有展示 AI 的能力
2. **过于刻意**：像面试一样一个问题一个问题地问，用户感觉在被审讯

### 我们的核心假设

**好的 onboarding 应该像「第一次见一个有趣的人」**——有新鲜感、有来有回、聊完之后觉得「这个人挺有意思的，而且真的能帮到我」。

这意味着 onboarding 必须同时满足两个看似矛盾的要求：
- **自然**：像朋友聊天，不像问卷
- **高效**：5-8 轮内完成所有目标，不能无限闲聊

「导演主导制」就是为了解决这个矛盾而设计的。

## 二、Onboarding 的定义与边界

### Onboarding 是什么

Onboarding 是用户与 AI 伙伴的「第一次对话」。它的产出是：
- AI 获得了一个名字和基本人设（由用户定义）
- AI 了解了用户的基本信息（职业、兴趣、当前关注的事）
- AI 展示了自己的行动力（主动提出帮忙做一件具体的事）
- 用户对 AI 的个性有了初步感知（幽默、好奇、直接）

### Onboarding 不是什么

| 不是              | 原因                                                          |
|-------------------|---------------------------------------------------------------|
| 不是注册流程      | 没有账号、没有表单、没有「下一步」按钮                        |
| 不是功能引导      | 不介绍功能列表、不教用户怎么用、不弹引导气泡                  |
| 不是性格测试      | 不通过问卷判断用户类型，而是在聊天中自然了解                  |
| 不是无限对话      | 有明确的结束条件（用户同意做一件事），不是开放式闲聊          |
| 不是一次性的      | Onboarding 收集的信息会持久化，影响后续所有对话               |

### 边界与限制

**当前方案的已知限制**：

1. **每次对话 2 次 LLM 调用**：导演 + 对话 AI，延迟和成本是单次调用的 2 倍。对话体验上可接受（用户有阅读时间），但规模化时需要考虑成本优化。

2. **导演的判断依赖 prompt engineering**：导演的目标推进、阶段判断完全靠 prompt 控制，没有硬编码的状态机。好处是灵活，风险是边界 case 可能判断不准（比如用户说了模棱两可的话，导演可能误判为「同意做事」）。

3. **人设一致性随对话增长衰减**：当 conversation history 变长，AI 的人设 prompt 在 context 中的比重下降，可能出现人设漂移。当前 onboarding 通常 5-8 轮，在可控范围内；但如果流程变长需要考虑 history 压缩。

4. **不支持中断恢复**：用户关闭页面后 onboarding 进度丢失，需要重新开始。当前是 prototype 阶段可以接受，产品化需要加状态持久化。

5. **单一语言**：当前 prompt 和 UI 都是中文，暂不支持多语言。

6. **无用户分群**：所有用户走同一套 onboarding 流程，没有根据用户类型（重度/轻度、年龄段、使用场景）做差异化。

## 三、项目目标

用户第一次打开 PandaClaw 时，通过一段自然的对话完成以下事项：

1. 建立「AI 伙伴」的人设感（不是工具，是一个有个性的角色）
2. 让用户给 AI 命名、定义形象
3. 了解用户基本信息（职业、兴趣、当前在忙什么）
4. 展示 AI 的行动力——主动提出帮忙做一件具体的事
5. 全程像微信聊天，不像问卷、不像客服

## 四、核心架构：导演主导制

### 架构图

```
用户消息
  │
  ▼
┌─────────────┐     5 行结构化输出
│  导演 LLM    │ ──────────────────┐
│ (temperature=0)                  │
│ max_tokens=200                   │
└─────────────┘                    │
                                   ▼
                          ┌─────────────────┐
                          │  对话 AI LLM     │
                          │ (temperature=0.8) │
                          │ max_tokens=500    │
                          └────────┬────────┘
                                   │
                                   ▼
                             多条气泡回复
```

### 分工原则

| 角色   | 职责                         | 不做什么               |
|--------|------------------------------|------------------------|
| 导演   | 决定这轮推进哪个目标         | 不管怎么说话、怎么接话 |
| 对话 AI | 接住用户的话 + 推进导演目标 | 不决定聊什么话题       |

### 导演输出格式（5 行）

```
completed: 1, 2
current_goal: 3
instruction: 推进形象配置
stage: onboarding
profile_update: name=小叶的虾
```

### 为什么用两个 LLM 调用？

单 LLM 方案的问题：AI 容易「聊嗨了忘记目标」或「为了赶目标忽略用户在说什么」。分成导演 + 对话 AI 后：
- 导演专注节奏控制，用 temperature=0 确保稳定判断
- 对话 AI 专注表达和人格，用 temperature=0.8 保持创造力
- 两者解耦，可以独立调优

## 五、Onboarding 流程

### 阶段 & 目标

```
icebreak ──→ onboarding ──→ done
  │              │              │
  破冰           命名           完成
  (1轮)         形象配置
               了解用户
               发起行动
```

| # | 目标     | 说明                                     | 节奏要求            |
|---|----------|------------------------------------------|---------------------|
| 1 | 破冰     | 初次见面感，结合时间 small talk           | 1 轮，用户回复后进入 onboarding |
| 2 | 命名     | 引导用户给 AI 取名字                     | 前 3 轮内推进       |
| 3 | 形象配置 | 性别、年龄、风格——用户参与定义           | 可与命名合并        |
| 4 | 了解用户 | 职业/背景、兴趣、在忙什么               | 不超过 3 轮         |
| 5 | 发起行动 | 根据用户提到的事，直接 offer 具体选项    | 一旦有线索就推      |
| 6 | 完成     | 用户同意做事                             | 触发 summary        |

### 结束判断

用户同意做某件事 → 导演输出 `stage: done` → 前端弹出 summary 面板 → onboarding 结束。

## 六、关键设计决策 & 迭代记录

### 1. 导演指令的注入位置

**问题**：导演指令放在 system prompt 末尾，AI 经常忽略。

**尝试过的方案**：
- system prompt 末尾 → AI 忽略率高
- system prompt 中加粗/强调语气 → 略有改善但不稳定
- **最终方案：作为最后一条 user message 注入** → AI 执行率显著提升

**原理**：LLM 对最近的 user message 注意力最高。把 `[导演指令]` 放在对话历史之后、AI 回复之前，成了 AI 回复前看到的最后一条内容。

```javascript
// server.js — buildMessages()
function buildMessages(conversationHistory, instruction) {
  const messages = [
    { role: 'system', content: systemPrompt + timeContext },
    ...conversationHistory
  ];
  if (instruction) {
    messages.push({ role: 'user', content: `[导演指令] ${instruction}` });
  }
  return messages;
}
```

### 2. 导演指令的长度控制

**问题**：导演输出太长时（如「用户提到了虾，这是一个很好的切入点，可以从这里引导到命名话题」），AI 反而更容易不执行。

**方案**：限制 instruction ≤ 15 个字，只写目标，不写策略。
- 好的 instruction：`推进命名，让用户取名`
- 坏的 instruction：`用户提到了虾，可以从这个话题引导到命名环节，建议先表达好奇`

### 3. AI 自主推理 vs 导演控制的平衡

**问题**：导演指令太具体 → AI 失去个性和推理能力（比如不会从「小叶的虾」猜到用户叫小叶）。导演指令太模糊 → AI 聊嗨了不推进目标。

**最终平衡**：
- 导演：只给目标方向（what），不给执行策略（how）
- AI system prompt：「用你的最后一条消息推进目标，前面可以接话」
- 效果：AI 前 1-2 条消息自由发挥（接话、猜测、幽默），最后一条消息推进目标

### 4. 动态 profile 更新

**机制**：导演从对话中提取用户对 AI 的定义（名字、性别、风格等），通过 `profile_update` 字段传递给前端。

```
profile_update: name=小叶的虾
profile_update: gender=女,age=20出头,style=可爱
profile_update: 无
```

**前端响应**：
- 名字更新 → 页面左上角 h1 实时变更
- 风格更新 → 头像球体颜色变化（渐变色映射）

### 5. 时间感知

AI 感知当前时间（凌晨/上午/下午/周几/几月），自然融入对话：
- 晚上 11 点打开 → 「夜猫子啊」
- 周一上午 → 「周一一大早」

## 七、技术栈

| 组件     | 技术                        |
|----------|-----------------------------|
| 后端     | Express.js (Node)           |
| LLM      | Claude Sonnet 4.6 via LiteLLM proxy |
| 配置     | YAML（prompts.yaml），支持热重载 |
| 前端     | 原生 HTML/CSS/JS，无框架   |
| 消息格式 | `|||` 分隔的多气泡消息      |

## 八、当前状态 & 已验证效果

- 导演指令执行率：高（注入位置优化后）
- AI 个性保持：好（幽默感、好奇心、大胆猜测）
- Onboarding 节奏：可控（导演节奏规则 + 阶段递进）
- 命名 → 页面更新：实现
- 形象配置 → 头像变化：实现
- Summary 面板：完成时自动生成对话总结

## 九、后续可探索的方向

### 产品层面

| 方向                 | 说明                                                         |
|----------------------|--------------------------------------------------------------|
| AI 形象可视化        | 从纯色球体升级为生成式头像（根据用户描述的风格生成）         |
| Onboarding 后的衔接  | 完成 onboarding 后，AI 直接开始帮用户做第一件事              |
| 记忆持久化           | 将 onboarding 收集的信息存储，后续对话中使用                 |
| 多轮 onboarding      | 不一次做完，分 2-3 次自然地了解用户                          |
| A/B 测试框架         | 不同人设 / 不同目标顺序的对比实验                            |

### 技术层面

| 方向                  | 说明                                                        |
|-----------------------|-------------------------------------------------------------|
| 导演 prompt 参数化    | 目标清单、节奏规则做成可配置项，方便实验                    |
| 流式输出              | 当前是完整回复，可改为 SSE 流式，提升体感速度               |
| 导演 + 对话合并调用   | 探索单次调用 structured output 同时完成导演和对话            |
| 前端框架化            | 当前原生 JS，如果功能扩展考虑迁移 React/Vue                 |
| 多模态               | 语音输入/输出，让初次见面更自然                              |

### 人设层面

| 方向              | 说明                                                          |
|-------------------|---------------------------------------------------------------|
| 人设预设模板      | 提供几种预设人格（活泼/沉稳/毒舌），用户可选或自定义         |
| 动态人设进化      | AI 的人设不在 onboarding 固定，随着长期互动逐渐变化           |
| 关系深度模型      | 根据对话轮数/频率/话题深度，解锁不同的 AI 表达层次           |
